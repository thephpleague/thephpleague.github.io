The League of Extraordinary PHP Packages

Just some lovely PHP code really.

The League of Extraordinary Packages is a group of developers who have banded together to build some pretty cool stuff. The aim is to build out solid, stable, well tested packages with modern standards, respecting the work of the PHP-FIG, following the best-practices put forward by PHP The "Right" Way and distributing code through Packagist and Composer.

More Information

We're doing this instead of releasing code under our personal accounts for a few reasons, but the main reason is this: A problem shared is a problem halved. Working together we take care of business quicker, can get more feedback and respond to issues and pull requests faster.

There are no plans to reinvent any wheels, unless those wheels are old, broken, unsafe or horrible to use. If the code cannot be installed with Composer, has an API written with BizZaroCapS, doesn't have a single unit-test, is actually broken or the lead developer has abandoned the project, then that is a problem which can most likely be solved with a fresh start, and we'll be on the case to make it as awesome as possible.

Members

If this sounds fun and you'd like to join the group you can definitely do that. This is not some stuffy old-boys club where you have to be internet famous or an old friend, you just need to bring over (or start) at least one package that either complies with the rules, or intends to comply with the rules by the next major version.

This group is about packages, and not about the people. To keep that clear - and to make sure people are joining for the right reasons - no members list will be posted, but we'll let you know who is in charge of packages so you can poke them with feedback.

This is not the FIG

We are not here to tell you what to do or how to do it - and contrary to popular belief neither are they. This is an open-source project started by a few friends who wanted a fun vendor name and will continue to grow as we release more and more awesome packages.

Our Definition of Quality

Ask 100 developers what defines an awesome PHP package and you'll get a lot of different answers. That also sounds like a really long and boring task, so we've decided to come up with a list of rules that make a package awesome:

  1. Follow PSR-2, we use League\ as our PSR-0 namespace.
  2. List on Packagist, we list with league\ as the vendor namespace.
  3. Shove code in a src/ folder.
  4. Write unit tests. Aim for at least 80% coverage for v1.0.
  5. DocBlock all the things.
  6. Semantic versioning must be used to manage version numbers.
  7. An extensive README.
    • Installation. Example
    • Usage. Example
    • Requirements
    • Contribution instructions (link to a CONTRIBUTING.md)
    • Framework Integration (Service Providers for Silex and/or Laravel 4 at minimum)
    • Author
    • License (MIT or DBAD)
    • Acknowledgements
  8. Include Travis-CI widget in README.
  9. Include Packagist widgets. Example.
